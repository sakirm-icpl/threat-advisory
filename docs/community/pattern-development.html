Pattern Development Guide

This guide is for experienced contributors who want to create advanced patterns or improve existing ones. It covers best practices, advanced techniques, and common pitfalls to avoid.

Advanced Pattern Design

Understanding Capture Groups

Capture groups are crucial for extracting version information. The version_group field specifies which capture group contains the version string. Make sure this is correctly set and tested.

Complex Pattern Structures

For complex software that reports versions in multiple ways, consider creating multiple patterns in a single file:

{
  "patterns": [
    {
      "name": "Apache HTTPD - Server Header",
      "category": "web",
      "pattern": "Server: Apache/([\\d.]+)",
      "vendor": "Apache",
      "product": "HTTPD",
      "version_group": 1,
      "priority": 180,
      "confidence": 0.9,
      "metadata": {
        // ... metadata ...
      }
    },
    {
      "name": "Apache HTTPD - X-Powered-By Header",
      "category": "web",
      "pattern": "X-Powered-By: Apache/([\\d.]+)",
      "vendor": "Apache",
      "product": "HTTPD",
      "version_group": 1,
      "priority": 160,
      "confidence": 0.85,
      "metadata": {
        // ... metadata ...
      }
    }
  ]
}

Regex Best Practices

Performance Considerations

Patterns should be efficient to avoid performance issues in scanning tools:

  • Avoid excessive backtracking
  • Use specific character classes when possible
  • Anchor patterns when appropriate
  • Use non-capturing groups for elements you don't need to extract

Common Pitfalls

Avoid these common regex mistakes:

  • Overly broad patterns that match unrelated text
  • Not escaping special characters properly
  • Incorrect capture group numbering
  • Not accounting for whitespace variations

Advanced Testing Techniques

Negative Test Cases

Include test cases that should NOT match your pattern to ensure specificity:

"test_cases": [
  {
    "input": "Server: Apache/2.4.41 (Ubuntu)",
    "expected_version": "2.4.41"
  },
  {
    "input": "Server: Nginx/1.18.0"
    // No expected_version means this should NOT match
  }
]

Edge Case Testing

Test edge cases such as:

  • Pre-release versions (alpha, beta, RC)
  • Non-standard version formats
  • Very long version strings
  • Unicode characters in version strings

Pattern Optimization

Priority and Confidence Tuning

Fine-tune these values based on:

  • Real-world testing results
  • False positive/negative rates
  • Specificity of the pattern
  • Reliability of the source

Working with the Validation Tools

Automated Validation

Use our validation tools to check your patterns:

# Validate a single pattern
python tools/validate-pattern.py patterns/web/apache.json

# Validate all patterns
python tools/validate-all-patterns.py

Continuous Integration

All patterns are automatically validated when submitted via pull request. Make sure your patterns pass validation before submitting.

Collaboration and Review

Peer Review Process

All pattern submissions go through peer review:

  1. Automated validation checks
  2. Manual review by maintainers
  3. Testing against real-world samples
  4. Feedback and revision requests
  5. Approval and merge

Incorporating Feedback

Be open to feedback and willing to iterate on your patterns. Reviewers may suggest improvements to accuracy, performance, or clarity.

Advanced Contribution Opportunities

  • Develop new validation tools or enhance existing ones
  • Create pattern analysis tools to identify potential improvements
  • Build integrations with popular security scanning frameworks
  • Research and document version detection techniques for new software categories
