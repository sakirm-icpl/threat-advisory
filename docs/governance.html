Project Governance

How the Version Detection Pattern Database project is managed

This document outlines the governance model for the Version Detection Pattern Database project.

Roles and Responsibilities

Our project has several roles with different responsibilities:

Contributors

Community members who contribute to the project in various ways:

  • Submitting patterns and improvements
  • Improving documentation
  • Enhancing tools and scripts
  • Participating in discussions

Maintainers

Responsible for the overall health and direction of the project:

  • Reviewing and merging pull requests
  • Managing issues and feature requests
  • Ensuring quality and consistency
  • Maintaining documentation

Project Lead

Has additional responsibilities:

  • Setting project vision and strategy
  • Coordinating with maintainers
  • Representing the project publicly
  • Making final decisions

Decision Making

We strive to make decisions through consensus among maintainers and active contributors. This means:

  • Discussing issues openly and thoroughly
  • Seeking input from all stakeholders
  • Finding solutions that most people can accept
  • Documenting decisions for future reference

If consensus cannot be reached on a significant issue, maintainers may call for a vote. Voting is used sparingly and only for critical decisions.

Contribution Process

All participants must follow our Code of Conduct. Violations will be addressed according to the enforcement procedures outlined in that document.

Pull Request Process

  • All contributions must go through the pull request process
  • Pull requests must be reviewed by at least one maintainer
  • Significant changes should be discussed in issues first
  • Maintainers have the right to request changes or reject contributions

Quality Standards

  • Contributions must follow our templates and guidelines
  • Data must be accurate and well-documented
  • Documentation should be clear and comprehensive
  • Code should be clean and well-commented

Release Process

Versioning

We follow Semantic Versioning (SemVer) for all releases.

Release Cadence

  • Major releases: As needed for significant changes
  • Minor releases: Quarterly or as needed for new features
  • Patch releases: As needed for bug fixes and security updates

Conflict Resolution

The Code of Conduct Committee handles violations of our Code of Conduct.

Technical Decisions

Technical disagreements should be resolved through:

  1. Discussion among involved parties
  2. Mediation by maintainers if needed
  3. Final decision by the Project Lead if consensus cannot be reached

Changes to Governance

This governance document can be changed by:

  1. Opening a GitHub issue to propose changes
  2. Discussing the proposal with the community
  3. Getting approval from a majority of maintainers
  4. Merging the changes after review

Significant changes require broader community input.
